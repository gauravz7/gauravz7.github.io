How we made startup research 3× faster — and more repeatable — by not asking the model to do everything

Collapsing a 7-stage agent chain into a 5-stage fan-out, splitting LLM extraction from deterministic scoring, and hand-rolling a resilient harness — with the tradeoffs named.

gauravz7 · engineering

We built an internal tool that turns a company name into a scored, sourced sales brief: funding signals, AI stack, competitive exposure, a paste-ready first email. The first version worked but took two to three minutes per company — too slow to use while a sales conversation is still live, and its scores drifted run to run because a language model was doing the scoring.

The rewrite (Startup Radar) lands a single company in 45–90 seconds and produces the same score every time for the same inputs. The speed came from restructuring the pipeline. The repeatability came from a design decision that matters more: we stopped asking the model to do the parts that code does better. This is what we changed and where it bit us.

The shape, end to end

One Cloud Run container serves the Next.js frontend and the FastAPI backend. A request becomes a job, the orchestrator runs the pipeline behind a shared throttle, and every model call, web fetch, and score writes through to Firestore so reports survive scale-to-zero. Progress streams back to the browser over SSE the whole time.

The mental model: LLMs extract, code decides

The original tool ran a seven-stage sequential chain of LLM calls, and the model was responsible for everything — including the final priority score and the "who are they competing with" call. That's two jobs a language model is bad at. Scoring needs to be reproducible and auditable; the model gave us neither. Detecting a competitor's name in a blob of text is a string-matching problem; handing it to an LLM adds latency, cost, and a chance of hallucination.

So we drew a hard line. The model does what only it can — disambiguate an entity, read a screenshot, synthesize prose. Deterministic Python does everything with a right answer. Concretely, the pipeline is five stages, and only three of them call Gemini for judgment:

#StageWho does the work
1DisambiguateGemini (grounded) — resolve "BOLT" to a real legal entity
2Asset FindGemini × 8 parallel grounded searches — site, GitHub, app stores, funding, hiring, news, social
3Deep InspectPlaywright fetch + screenshots → Gemini reads the images and writes a positioning paragraph
4AI IntelGemini synthesizes; then deterministic regex scores competitive exposure
5Pitch PlaybookGemini writes the email, hooks, objections

Deterministic scoring, so the number means something

The priority score is a pure function of extracted fields, capped at 100:

funding_score = min(50, latest_funding_usd / 1_000_000)
ai_fit_score  = 25 if gen_ai_native or a GenAI product category
recency_score = 15 / 8 / 3 by how recent the latest momentum signal is (90/180/365d)
hiring_score  = min(10, ai_ml_open_roles)

Tiers fall out of thresholds: SUPER_HIGH needs ≥ $5M raised, a round within 180 days, and a total ≥ 60; otherwise HIGH ≥ 50, MEDIUM ≥ 25, LOW below.

The payoff isn't accuracy in the abstract — it's that the same company scores identically on every run, a salesperson can see why it scored 62, and we can change the rubric by editing a formula instead of re-tuning a prompt and re-checking every past report. The cost is honest: the score is only as good as the fields the model extracted. If Gemini misses a funding round, deterministic math will confidently score the miss. We pushed the hard problem (extraction) to the model and kept the easy-but-must-be-right problem (arithmetic) in code — but we didn't make extraction any easier.

Competitive detection is regex, not a judgment call

Stage 4's most useful output is "which third-party AI vendors a company is already using" — the competitive displacement angle. That's ~30 weighted patterns run over every piece of collected text:

(r"\bopenai\b|\bgpt-?[345]\b|\bchat-?gpt\b", "OpenAI / GPT-*", 35),
(r"\banthropic\b|\bclaude(?:-3)?\b",          "Anthropic / Claude", 35),
(r"\bamazon\s+bedrock\b",                     "AWS Bedrock", 30),
(r"\belevenlabs\b",                           "ElevenLabs (voice)", 22),
(r"\bcursor\.(?:com|so)\b",                   "Cursor", 22),
# …and ~25 more, each with a weight

A second, smaller pattern set scans for our own platform's footprint and lowers the exposure score — a company already on it isn't a swap target. Every match emits an evidence string, so the report shows the exact text that triggered "AWS Bedrock", not an unsourced assertion. This is faster, free, and auditable in a way an LLM classifier is not. It's also brittle by nature: a vendor we haven't added a pattern for is invisible, so the list is a living file, not a finished one.

The concurrency bug you only hit after you fan out

Speed came from parallelism, and parallelism is where it got interesting. Inside one company, stage 2 fires 8 grounded searches at once with asyncio.gather. Across a batch, the orchestrator runs 5 companies concurrently behind a Semaphore. Multiply it out: up to 40 in-flight Gemini calls, plus the per-company synthesis and vision calls on top.

Vertex started returning 429s well below the documented quota. The reason took a while to find: the grounded-search backend, the chat backend, and the multimodal backend draw from shared underlying pools, so "requests per minute" on paper isn't the ceiling you actually hit when you spread load across all three at once. Per-stage rate limiting didn't help, because no single stage was over its limit.

The fix is one process-wide throttle every Gemini call passes through, combining a concurrency cap and a QPS token bucket — and the acquisition order matters:

async def __aenter__(self):
    await gemini_semaphore().acquire()      # get a concurrency slot FIRST
    try:
        await gemini_limiter().__aenter__() # THEN spend a QPS token
    except BaseException:
        gemini_semaphore().release()
        raise

Grab the semaphore first, spend a rate-limit token second. Reverse it and you burn QPS tokens on calls that are still queued for a concurrency slot — you rate-limit work that isn't even running yet. Same two limits, wrong order, and throughput drops. Underneath, every call is still wrapped in exponential backoff on 429/503, because a shared throttle reduces contention but doesn't make a distributed quota perfectly predictable.

The agent is five prompts; the harness is everything else

The original tool used a framework's sequential-agent primitive: declare seven stages, let the framework run them in order. It was quick to stand up and it's why we shipped a v1 at all. It's also why v1 was slow and fragile — the framework owned the control flow, so concurrency, per-stage failure handling, timeouts, and mid-run recovery were whatever the framework decided, which was "run them one at a time, and if one throws, the run dies."

For v2 we kept the agent small — five prompts and their structured-output schemas — and hand-wrote the harness around them. That's the trade at the center of this project: we gave up the framework's convenience to own the failure modes. The agent is the easy part. The harness is where the engineering lives, and it's four decisions, each with a cost.

Graceful degradation over all-or-nothing. In v1, a transient hiccup in stage 4 lost the whole company. Now stages 3–5 catch their own exceptions and emit a partial result — an empty intel object, a "pitch skipped" note — so the report writes with whatever the earlier stages produced:

try:
    intel = await ai_intel(disambig, assets, inspection)
except Exception as e:
    intel = AISalesIntel(deep_search_narrative=f"intel synthesis skipped: {e}")
    await event(4, "ai_intel", "done", f"degraded: {str(e)[:160]}")

The cost is real and we don't love it: a degraded report can look complete. We mark needs_review and surface the "degraded" note in the trace, but a rushed reader can still miss that stage 4 was empty. We decided a partial brief the salesperson can act on beats a hard failure that gives them nothing — but that's a judgment call, not a free win.

A hard per-company timeout. Fan-out has a failure mode you don't see until you're running batches: a single Vertex call stuck in its retry loop holds a concurrency slot forever, silently shrinking the parallel pool one company at a time. Every company is wrapped in asyncio.wait_for(PER_COMPANY_TIMEOUT_S) (900s). The tradeoff is picking the number: too low and you kill legitimately slow companies in a big batch; too high and a genuine hang wastes a slot for fifteen minutes. We landed on 900s after 300s started killing real work — measured, not guessed, and still just a compromise.

Two independent layers against 429s. The shared throttle from the last section prevents most rate-limit errors; the retry helper recovers from the ones that slip through, with exponential backoff and jitter (1s, 2s, 4s, 8s, 16s, ~30s worst case). The deliberate part is what we don't retry — only 429 and 5xx are treated as transient. A 400, 401, or 404 surfaces immediately, because retrying a wrong-model or bad-auth error just hides a real bug behind thirty seconds of backoff. Belt and suspenders on transient failures; fail-fast on everything else.

Restartable batches, at one write per company. Cloud Run scales to zero and recycles instances, so a batch has to survive its host dying. After every company we persist the batch record — full company list plus a completion counter — to Firestore, and a startup hook picks up unfinished batches on the next boot. We also chunk the batch and gc.collect() between chunks rather than pre-creating 8,000 coroutines up front, which would OOM the container. Both cost something: an extra Firestore write per company, and a small barrier between chunks where the slowest company stalls its chunk. For a job that runs for hours across restarts, we'll take both.

The through-line: a framework's sequential agent is the right call to prove the idea, and the wrong call once uptime, latency, and partial-failure behavior are the product. Moving the control flow into our own harness cost us the framework's guardrails and made us write our own — but every failure mode above is now a decision we made on purpose instead of one we inherited.

Structured output, so failures are loud

The old tool asked for free text and pulled fields out with regex against markers like [SCORES]. It worked until the model rephrased something and a field came back empty — silently, then wrong, downstream. The rewrite uses Gemini's structured-output mode with Pydantic v2 models, validated at the boundary. A missing field now fails at the source instead of surfacing as a blank cell in someone's spreadsheet three steps later. This bought us zero speed and a lot of trust, which is the trade we'd make again.

The cache makes the 23K-row run restartable

Both tools share one Firestore cache, keyed by a normalized company name, and we kept that key byte-compatible on purpose. When we scored a 23,176-company partner list into a separate collection, the cache short-circuit made restarts free: a worker dies at row 12,000, we re-run the same script, it skips everything already scored and continues. The batch scripts also hard-refuse to write to the production companies collection — isolation enforced in code, not in a runbook, because runbooks get skipped at 2 a.m. The CSV export stays byte-identical to the legacy tool's column order, so existing sheets never broke during the migration.

What we'd tell someone building the same thing

The faster model everyone assumes was the fix was the least of it. The wins were architectural, and they generalize:

  1. Split the work by what has a right answer. Let the model extract and synthesize; keep scoring and matching in deterministic code you can read, test, and reproduce.
  2. Throttle shared model backends as one pool, and mind the acquisition order. Fan-out exposes quota behavior that per-stage limits can't see; take the concurrency slot before the rate-limit token.
  3. Make outputs structured and cache keys stable. Structured output turns silent failures loud; a stable key turns restarts and migrations into non-events.

If your research agent is too slow and too inconsistent, those are usually the same bug: too much asked of the model, in too long a line. Move the deterministic parts into code, fan out the I/O, and throttle it as one — then a smarter prompt is a bonus, not the load-bearing wall.
