Build your own Household Buddy

A WhatsApp assistant that turns your family's messages, photos, and school emails into one shared to-do list — and reminds you twice a day. This is the complete, step-by-step build.

Photos toonotices, invites

By the end you'll have a bot each parent messages privately on WhatsApp, backed by one shared memory. Either parent can file a task ("Leo's party Saturday 1:30"), snap a photo of a school notice, remove something, or just ask "what's on this week?" — and a scheduled digest lands on both phones every morning and evening.

Everything here is real and battle-tested — including the handful of traps that will otherwise cost you an afternoon each. Follow the parts in order.

1 Scaffold the app

Create a small Node project. The dependencies are deliberately few:

mkdir household-buddy && cd household-buddy
npm init -y
npm pkg set type=module
npm install express @google/genai @google-cloud/firestore luxon

Organise the source into focused files under src/:

FileIts one job
server.jsExpress app: the /webhook and /cron/digest routes
handler.jsWhat to do with an inbound message (the conversation logic)
brain.jsEvery Gemini call — extract tasks, reply, build the digest
db.jsFirestore reads/writes, all scoped by household
whatsapp.jsThin wrapper over Meta's Graph API (send text, send template, fetch media)
seed.jsCreates your household, parents, and kids on first boot

Add start scripts to package.json:

"scripts": {
  "start": "node --env-file-if-exists=.env src/server.js",
  "dev": "node --env-file-if-exists=.env --watch src/server.js"
}

The core inbound route stays tiny — acknowledge Meta within 5 seconds, then work asynchronously, or Meta retries and you double-process:

// server.js — the entry point
app.post('/webhook', async (req, res) => {
  res.sendStatus(200);                     // ack FIRST
  try {
    const msg = req.body.entry?.[0]?.changes?.[0]?.value?.messages?.[0];
    if (msg) await handleInboundMessage(msg, { send: sendText, fetchMedia });
  } catch (e) { console.error('webhook error', e); }
});

2 Get your Gemini API key

At aistudio.google.com (Google AI Studio), create an API key and keep it for your .env (Part 5). All model calls live in brain.js and share one grounding snapshot of household state (kids, members, open tasks) pulled fresh from Firestore on every call — that's what makes the memory feel shared.

A single extraction call turns a message into structured tasks:

// brain.js — extract tasks as strict JSON
const resp = await genai.models.generateContent({
  model: 'gemini-3.1-flash-lite',
  contents,                                  // text and/or a base64 image part
  config: {
    systemInstruction: SYSTEM_PROMPT,
    thinkingConfig: { thinkingBudget: 0 },   // ← see the gotcha below. Do not skip.
    responseMimeType: 'application/json'       // enforce clean JSON
  }
});
const text = resp.text;
Gotcha that will cost you an afternoon

Gemini models also emit a hidden thinking block by default. On these bounded formatting/extraction calls it just adds latency and can crowd a tight output budget, so disable it with thinkingConfig: { thinkingBudget: 0 } on a Flash-class model. (The same trap bit this project on a previous model, where it left calls returning no text at all — worth checking on any LLM you swap in.)

3 Set up WhatsApp Cloud API

This is the fiddliest part. Take it slowly.

3a · Create the app and add WhatsApp

  1. Go to developers.facebook.com → My Apps → Create App → type Business.
  2. Add the WhatsApp product. Meta gives you a free test phone number.
  3. From the WhatsApp → API Setup panel, copy two IDs you'll need: the Phone Number ID and the WhatsApp Business Account (WABA) ID.

3b · Allowlist your recipients

A test number can only message numbers you explicitly add. In the API Setup panel, add each parent's WhatsApp number (with country code) to the recipient list and confirm the code they receive.

3c · Get a token that never expires

The temporary token in the dashboard dies in 24 hours — useless for a always-on bot. Create a permanent one:

  1. Business Settings → Users → System Users → create one (Admin).
  2. Assign assets → add your WhatsApp Account to that system user with full control. (Skip this and the token is valid but powerless.)
  3. Generate token → select your app → scopes whatsapp_business_messaging and whatsapp_business_management → set expiry to Never.
Verify the token actually works

Before moving on, confirm it can see your number:

curl "https://graph.facebook.com/v21.0/<PHONE_NUMBER_ID>?access_token=<TOKEN>"

4 Google Cloud project + Firestore

Cloud Run's disk is ephemeral — anything written locally vanishes on redeploy. Use a real managed database. (Don't try to mount a bucket and run SQLite on it: SQLite's small out-of-order writes corrupt over a bucket's FUSE layer.)

gcloud config set project YOUR_PROJECT_ID
gcloud services enable run.googleapis.com firestore.googleapis.com \
  cloudscheduler.googleapis.com cloudbuild.googleapis.com

# Create Firestore in native mode (pick a region near you)
gcloud firestore databases create --location=asia-southeast1

Your db.js talks to Firestore through Application Default Credentials — on Cloud Run that's automatic, no key file. All queries are scoped by household_id, so there's never cross-household visibility.

The identity rule that trips everyone

A member's phone number is the primary key and must match WhatsApp's wa_id exactly — country code, digits only, no +, spaces, or dashes. seed.js strips non-digits, but get this wrong and inbound messages won't match any member.

seed.js runs on every boot (imported for its side effect in server.js) and idempotently creates the household, both parents, and the kids from these vars.

6 Deploy to Cloud Run

Deploy straight from source. Keep --min-instances 1 so the bot is always warm and never cold-starts past Meta's timeout:

gcloud run deploy household-buddy --source . \
  --region asia-southeast1 --allow-unauthenticated --min-instances 1 \
  --set-env-vars "$(grep -v '^#' .env | grep . | paste -sd, -)"

When it finishes it prints a Service URL like https://household-buddy-xxxx.run.app. Save it — it's your webhook base. Sanity-check it:

curl https://your-service.run.app/health   # → ok

7 Connect the webhook

Two separate wirings are needed — miss the second and messages arrive in Meta's dashboard but never reach your app.

7a · Set the callback URL

WhatsApp → Configuration → Edit. Callback URL = https://your-service.run.app/webhook; Verify token = the WA_VERIFY_TOKEN you chose. Your GET /webhook handles the handshake. Then subscribe to the messages field.

7b · Subscribe your app to the WABA

This is the step people miss. The app must be explicitly subscribed to the business account:

curl -X POST \
  "https://graph.facebook.com/v21.0/<WABA_ID>/subscribed_apps" \
  -H "Authorization: Bearer <TOKEN>"

Now message the bot from an allowlisted number. You should get a reply. 🎉 (Remember: replies only work inside a 24-hour window after the user's last message — the next part fixes proactive delivery.)

8 The twice-daily digest

A POST /cron/digest route (guarded by DIGEST_SECRET) pulls open tasks, splits them into Today and Next 7 days, and messages both parents. Two Cloud Scheduler jobs fire it:

URL=https://your-service.run.app/cron/digest?household=smith
for t in "morning 0 6" "evening 0 18"; do set -- $t
  gcloud scheduler jobs create http buddy-digest-$1 \
    --location=asia-southeast1 --schedule="$2 $3 * * *" \
    --time-zone=Asia/Singapore --http-method=POST --uri="$URL" \
    --headers="x-digest-secret=your-digest-secret"
done
The most important lesson in this whole guide

WhatsApp only lets a business send free-form text within 24 hours of the user's last message. A scheduled digest is unprompted — so once your parents stop chatting for a day, every free-form digest is silently accepted by the API but never delivered. You cannot fix this with a keep-alive ping: only the user messaging you reopens the window. The real fix is an approved template, which delivers anytime.

8a · Create & submit the template

Template variables can't contain newlines, so structure lives in the fixed body and each variable is one tidy line:

curl -X POST "https://graph.facebook.com/v21.0/<WABA_ID>/message_templates" \
 -H "Authorization: Bearer <TOKEN>" -H "Content-Type: application/json" \
 -d '{
   "name":"daily_digest","language":"en","category":"UTILITY",
   "components":[{"type":"BODY",
     "text":"🏠 Household digest — {{1}}\n\n📌 Today: {{2}}\n\n🗓️ Next 7 days: {{3}}",
     "example":{"body_text":[["Thu 16 Jul","Pay fees (Leo) · Buy milk","Camp Mon · Fees Fri"]]}
   }]
 }'

Utility templates usually clear review within the hour. Poll until APPROVED:

curl "https://graph.facebook.com/v21.0/<WABA_ID>/message_templates?name=daily_digest&access_token=<TOKEN>"

8b · Send the digest as that template

// whatsapp.js — delivers regardless of the 24h window
await fetch(`${GRAPH}/${PHONE_NUMBER_ID}/messages`, {
  method: 'POST',
  headers: { Authorization: `Bearer ${TOKEN}`, 'Content-Type': 'application/json' },
  body: JSON.stringify({
    messaging_product: 'whatsapp', to, type: 'template',
    template: { name: 'daily_digest', language: { code: 'en' },
      components: [{ type: 'body', parameters: [date, today, week].map(t => ({ type: 'text', text: t })) }] }
  })
});

Build date, today, and week deterministically (no LLM) so they're always single-line and never empty — Meta rejects blank variables. Interactive chat keeps using free-form text; only the proactive digest needs the template.

9 Mine school email — optional

Schools send fees, forms, and dates by email. Rather than wrestle Gmail's OAuth verification, use a tiny Google Apps Script that runs inside your own account on a weekly trigger and POSTs new school mail to a secured /ingest/email endpoint.

// Apps Script (script.google.com) — runs weekly, AS you, no OAuth app to verify
function scanSchoolMail() {
  const q = '(from:school.edu OR "Fee Invoice") newer_than:8d -label:buddy-done';
  for (const thread of GmailApp.search(q, 0, 50)) {
    const m = thread.getMessages().pop();
    UrlFetchApp.fetch('https://your-service.run.app/ingest/email', {
      method: 'post', contentType: 'application/json',
      headers: { 'x-ingest-secret': SECRET },
      payload: JSON.stringify({ messageId: m.getId(), from: m.getFrom(),
        subject: m.getSubject(), body: m.getPlainBody() })
    });
    thread.addLabel(doneLabel);
  }
}

Two things that matter more than you'd expect:

  • Dedup before the model. Record each Gmail messageId you've handled and skip it — so the model only ever runs on genuinely new mail.
  • Invoices often don't come from the school's domain. Many schools bill through a vendor (e.g. NetSuite). A naïve from:school.edu filter misses exactly the emails you most want — match a subject phrase too.

Set the weekly trigger once (Triggers → time-driven → weekly), and set the Apps Script project's timezone so "Sunday morning" means your morning.

// under the hood

How the bot thinks: no agent, just prompts + a state machine

There's no autonomous agent here — no tool-calling loop that decides its own next step, no multi-agent framework. The model is used as a stateless function (text in → JSON or a reply out), and ordinary JavaScript in handler.js decides which prompt to call. That keeps the bot predictable, cheap, and testable.

Every inbound message runs the same short decision tree, with the conversation's phase saved in Firestore between messages:

message in
 → conversation in progress?
     • "clarifying" → advanceClarification   // merge answer; maybe ask once more
     • "confirming" → resolveConfirmation    // yes / no / unclear
 → else proposeFromMessage:
     • remove / done request → update tasks, read back to both parents
     • vague new task        → ask ONE question    (→ clarifying)
     • clear new task        → composeConfirmation  (→ confirming)
     • a question / chit-chat → composeReply

Two other paths run independently and are just as un-agentic: the twice-daily digest (a scheduled cron route) and the weekly school-email scan (its own endpoint). Each is a single prompt call wrapped in a route.

The eight prompts

All the "intelligence" is a set of system prompts in brain.js, one per call. Every one is grounded the same way: right before the call, a fresh snapshot of household state — kids, members, open tasks — plus TODAY and the timezone is appended. That shared snapshot is both what makes the memory feel shared and the guardrail against invented tasks.

Understanding calls — return strict JSON:

  • Propose — the main brain: extract new tasks to add, and detect remove/done requests against the open list.
  • Decide — classify a confirmation reply as confirm / reject / unclear.
  • Clarify — merge the parent's answer into the draft; decide if one more question is warranted (capped at two).
  • School — from one forwarded email, extract only real actions; ignore newsletters and app-notification digests.

Voice calls — return prose in the "Buddy" persona:

  • Reply — conversational answers, grounded in shared state + recent messages.
  • Confirm — "I think this photo is a birthday invite — file it? (yes/no)".
  • Read-back — after filing, repeat exactly what was saved so mistakes get caught.
  • Digest — the free-form daily summary (the template version uses a deterministic builder, no model at all).

The extraction prompt is the load-bearing one — its rules, roughly verbatim:

Return ONLY a JSON object, no prose, no markdown fences:
{ "tasks":[…], "needs_clarification":bool, "question":str|null,
  "remove_titles":[…], "done_titles":[…] }

- Only genuinely actionable NEW items in "tasks".
- If the message is a QUESTION or chit-chat, return everything empty.
  Never invent a task from a question.
- REMOVAL/COMPLETION: only on an explicit remove/delete/done command,
  copy the matching open task's title VERBATIM. Never guess.
- Ask ONE clarifying question only when a task is too vague to be useful.
Grounding is the whole trick

None of these prompts are clever on their own. What makes answers correct is that each is handed your actual current list and today's date at call time — so "what's on this week?" and "remove the milk" resolve against real state, not the model's imagination.

The traps, collected

  • Disable thinking on bounded model calls, or they return empty text.
  • Proactive messages need a template — the 24-hour window blocks free-form, and no self-ping reopens it.
  • Assign the WhatsApp asset to your system user, or the permanent token is powerless.
  • Subscribe the app to the WABA (/subscribed_apps) — the webhook field alone isn't enough.
  • Phone numbers must match wa_id exactly: country code, digits only, no +.
  • Ack the webhook with 200 first, then process — or Meta retries and you double-file.
  • Cloud Run storage is ephemeral — use Firestore, not a file on a mounted bucket.
  • The test number is dev-only (allowlist + limits). For production, register a real business number — the same template code works.
